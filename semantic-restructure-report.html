왜 semantic을 다시 짰나 (진단)

참조 분석 결과, 계층 비율이 역피라미드였습니다 — 가장 두꺼워야 할 semantic이 가장 얇았습니다.

구체적 문제 3가지

  • 분류 축 혼재 — alert(의미), border(역할), brand(의미), elevation(효과)가 같은 층에 형제로 섞임.
  • 의미·역할 혼용 — alert와 border는 서로 직교하는 축인데 나란히 둬서 "alert와 border가 같이 있는" 이상한 구조.
  • 상태 불일치 — 공유 상태 토큰(common/alpha)이 있는데도 거의 안 쓰여, 484개 컴포넌트가 hover·selected 색을 각자 primitive에서 직접 골라 제각각.

설계 원칙 — 3개 축의 직교 분리

"무엇인가(역할) × 어떤 의미인가(의미) × 어떤 상태인가(상태)"를 한 이름에 섞지 않고 분리했습니다.

역할 Role
text · bg · border · icon
의미 Intent
status(success/error/warning/info/neutral) · brand
상태 State
hover · pressed · selected · focus-ring · disabled

이 분리로 "alert·border 혼용"은 의미×역할 교차(status/error/border)로 풀리고, "상태 불일치"는 공유 state 토큰으로 풀립니다.

state/ — 상태 불일치의 해법 (키스톤)

모든 컴포넌트가 같은 상태 토큰을 참조하게 만들어, "다른 컴포넌트여도 같은 상태는 같은 색"을 구조적으로 보장합니다. hover/pressed는 알파 오버레이 방식이라 어떤 배경 위에서도 일관됩니다.

state/selected→ chiptabsegmentedlistmenu …
state/focus-ring (orange/500)→ 모든 컴포넌트 포커스
한 곳(state/*)만 바꾸면 484개 컴포넌트의 상태색이 동시에 일관되게 갱신됩니다.

다크모드 구조 — 테마를 semantic에서 전환

primitive의 light 램프를 "절대 팔레트"로 보고, semantic의 dark 모드가 토큰마다 더 밝은/어두운 step을 가리키도록 했습니다. (primitive 모드 의존 없이 semantic 모드만 바꾸면 테마 전환)

  • 중립·브랜드 — 대칭 스케일이라 미러 step으로 정확히 반전 (예: text/primary 900↔50)
  • 알파 오버레이 — 어두운 배경에선 밝게 올림 (black → white), 딤/그림자는 black 유지
  • 기능색 — 16단계라 정확한 미러가 없어 대비에 맞는 밝은 step 선택 (미세조정 가능)

그룹별 상세 가이드 — 종류 · primitive · 적용 위치

각 토큰이 무엇이고, 어떤 primitive에서 왔으며, 실제로 어디에 적용해야 하는지 정리했습니다. (light 기준)

명칭 체계가 그룹마다 다른 이유 — 역할마다 "단계"의 개념이 다릅니다. text·icon은 강조 수준(primary→secondary→tertiary), bg는 표면 깊이(base→layer1~3), border는 시각적 굵기(subtle/default/strong)입니다. 그래서 같은 primary/secondary를 모든 그룹에 쓰지 않습니다. (icon은 text와 동일하게 primary/secondary/tertiary 3단계로 맞췄습니다)

브랜드 색은 역할별 토큰으로만 관리합니다: bg/brand(브랜드 배경) · text/brand(브랜드 글자) · border/brand · icon/brand · bg/brand-subtle(옅은 배경) · text/on-brand(브랜드 위 글자).

category/ 그룹 — 해체됨 (component → primitive 직접참조로 전환)

범주형 구분색(tag·date_picker 이벤트색)은 "의미"가 아니라 "구분"이고, tag·date_picker를 독립 관리하기로 해서 공유 category/ 그룹을 해체했습니다. 각 컴포넌트가 primitive를 직접 참조합니다 (예: tag/bg/red → light red/200 / dark red/900). light/dark 값은 컴포넌트 모드에 그대로 이식돼 테마는 유지됩니다.
원칙: 테마·일관성이 필요한 색은 semantic 경유, 의미 없는 컴포넌트 전용 장식·구분색은 primitive 직접참조 허용.
